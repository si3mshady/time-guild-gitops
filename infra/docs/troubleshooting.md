# Platform Troubleshooting & Diagnostics

This manual provides instructions for diagnosing and resolving common issues encountered when running the **Time Guild / AURA** trust layer platform in a k3s cluster environment.

---

## 1. 5-Minute Diagnostics Workflow (SRE Dashboard)

When an incident is triggered (e.g. payout failure alerts), SREs use this workflow to isolate the root cause in under 5 minutes:

```
                  ┌────────────────────────────────────────┐
                  │ Incident Alert triggered               │
                  └──────────────────┬─────────────────────┘
                                     │
                                     ▼
                  ┌────────────────────────────────────────┐

                                     │
                                     ▼
```

## 2. Common Issues & Solutions

### Issue A: PostgreSQL Connection Pool Saturation
*   **Symptom**: Grafana HTTP latency metrics show a steep climb ($>2\text{s}$ response times). Web logs reveal `database connection pool limit reached` or `timeout waiting for pool connection`.
*   **Root Cause**: Pods are creating too many direct connections to PostgreSQL, exceeding connection limits during Horizontal Pod Autoscaler (HPA) replica scaling.
*   **Solution**:
    1. Verify active connection counts: `kubectl exec -it pg-pool-svc -n timeguild-prod -- psql -c "SELECT COUNT(*) FROM pg_stat_activity;"`
    2. Adjust PgBouncer configurations in `values.yaml` to increase pool limits, or decrease application pod connection lifetimes.

### Issue B: Failed Stripe Payouts (Idempotency Retries)
*   **Symptom**: Alertmanager fires a `HighStripePayoutFailures` alert. Bookings remain stuck in `completed` without transitioning to `paid`.
*   **Root Cause**: The BullMQ background payout job is failing due to credentials mismatch, network timeout, or Stripe API rate limits.
*   **Solution**:
    1. Search Loki logs for transaction correlation IDs matching the failed booking:
       ```
       {app="timeguild"} |= "booking_bk_999"
       ```
    2. Check the active job queue state in Redis:
       ```bash
       kubectl exec -it deployment/timeguild-redis -n timeguild-prod -- redis-cli info keyspace
       ```
    3. Re-verify the idempotency key configuration in [trust-rules.ts](file:///home/si3mshady/time-guild/src/lib/trust-rules.ts) to verify that payout tasks are retried without generating duplicate transactions.

### Issue C: Invalid Webhook Signatures (HTTP 400 Bad Request)
*   **Symptom**: Stripe payments complete, but bookings do not update. Checkouts fail with `Webhook signature verification failed` error logs.
*   **Root Cause**: The `STRIPE_WEBHOOK_SECRET` key is missing, or does not match the secret key generated by the Stripe Developer Dashboard.
*   **Solution**:
    1. Retrieve the webhook secret from the Stripe dashboard.
    2. Update the External Secrets Operator source, or update the Kubernetes secret directly:
       ```bash
       kubectl edit secret timeguild-secrets -n timeguild-prod
       ```
       *(Note: Values must be base64 encoded when updated manually)*

### Issue D: Dynamic Subdomain Routing Failures (Ingress 404)
*   **Symptom**: Navigating to `app.timeguild.com` works, but navigating to `sarah.timeguild.com` returns an Ingress `404 Not Found` page.
*   **Root Cause**: Wildcard record properties are missing in your DNS registrar, or the ingress-nginx controller is not matching the wildcard hostname rules.
*   **Solution**:
    1. Verify the Ingress rule configurations: `kubectl get ingress -n timeguild-prod`.
    2. Ensure that wildcard record settings (`*.timeguild.com`) point to your ingress controller IP.
